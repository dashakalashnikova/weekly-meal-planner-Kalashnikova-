# Вимоги до вебзастосунку  
# “Планувальник харчування на тиждень”

## 1. Призначення системи

Вебзастосунок “Планувальник харчування на тиждень” призначений для складання, збереження, перегляду та редагування тижневого плану харчування користувача.

Система має допомагати користувачу:

- створювати список страв;
- додавати страви до плану харчування на конкретний день тижня;
- обирати тип прийому їжі: сніданок, обід, вечеря або перекус;
- автоматично формувати список продуктів на основі обраних страв;
- використовувати теги та фільтри для швидкого пошуку страв;
- перевіряти прості правила сумісності страв у межах одного дня;
- зберігати план харчування між сесіями в браузері.

## 2. Межі системи

Система є однокористувацьким вебзастосунком, що працює в браузері без обов’язкового серверного бекенду.

До обсягу системи входить:

- створення страв;
- редагування страв;
- видалення страв;
- додавання страв до тижневого плану;
- зміна або видалення страви з конкретного прийому їжі;
- пошук страв за назвою;
- фільтрація страв за тегами, типом прийому їжі та рівнем складності;
- автоматичне формування списку продуктів;
- перевірка простих правил сумісності;
- збереження даних локально в браузері;
- відновлення даних після повторного відкриття застосунку;
- скидання плану або повернення demo data.

Поза обсягом системи:

- реєстрація та авторизація користувачів;
- синхронізація між різними пристроями;
- робота кількох користувачів одночасно;
- інтеграція з магазинами, службами доставки або платіжними сервісами;
- підрахунок точної калорійності та нутрієнтів;
- медичні рекомендації щодо дієти;
- професійний підбір раціону лікарем або дієтологом;
- push-сповіщення або email-нагадування;
- збереження даних на сервері.

## 3. Роль користувача

Користувач:

Людина, яка хоче заздалегідь спланувати харчування на тиждень, швидко підібрати страви, побачити список потрібних продуктів і уникнути повторення або несумісного поєднання страв.

## 4. Основні сутності

### 4.1. Страва

Страва має такі атрибути:

- ID;
- назва;
- опис;
- тип прийому їжі;
- список інгредієнтів;
- теги;
- рівень складності;
- орієнтовний час приготування;
- позначка “улюблена страва”;
- дата створення;
- дата останнього редагування.

### 4.2. Інгредієнт

Інгредієнт має такі атрибути:

- назва продукту;
- кількість;
- одиниця вимірювання;
- категорія продукту.

### 4.3. День тижня

Допустимі значення:

- Monday;
- Tuesday;
- Wednesday;
- Thursday;
- Friday;
- Saturday;
- Sunday.

### 4.4. Тип прийому їжі

Допустимі значення:

- Breakfast;
- Lunch;
- Dinner;
- Snack.

### 4.5. Запис у плані харчування

Запис у плані харчування має такі атрибути:

- ID;
- день тижня;
- тип прийому їжі;
- ID обраної страви;
- дата створення або оновлення запису.

### 4.6. Тег

Тег використовується для групування та швидкого пошуку страв.

Приклади тегів:

- швидко;
- легко;
- вегетаріанське;
- білкове;
- солодке;
- без м’яса;
- з куркою;
- на роботу;
- домашнє;
- корисне.

## 5. Функціональні вимоги

### FR-01. Створення страви

Система повинна надавати користувачу форму для створення нової страви.

Під час створення страви користувач повинен мати можливість вказати:

- назву страви;
- опис;
- тип прийому їжі;
- список інгредієнтів;
- теги;
- рівень складності;
- орієнтовний час приготування;
- ознаку “улюблена страва”.

Після успішного створення:

- страва додається до каталогу страв;
- їй автоматично присвоюється унікальний ID;
- фіксується дата створення;
- страва стає доступною для додавання в тижневий план.

### FR-02. Валідація форми створення та редагування страви

Система повинна перевіряти коректність введених даних перед збереженням страви.

У разі невалідного вводу система повинна:

- не зберігати страву;
- показувати повідомлення про помилку біля відповідного поля або у верхній частині форми;
- залишати вже введені користувачем дані у формі, щоб їх можна було виправити.

### FR-03. Перегляд каталогу страв

Система повинна відображати список усіх створених страв.

Для кожної страви в каталозі мають відображатися:

- назва;
- тип прийому їжі;
- рівень складності;
- орієнтовний час приготування;
- теги;
- позначка “улюблена страва”, якщо вона встановлена.

### FR-04. Редагування страви

Система повинна дозволяти користувачу редагувати наявну страву.

Користувач повинен мати можливість змінювати:

- назву;
- опис;
- тип прийому їжі;
- список інгредієнтів;
- теги;
- рівень складності;
- орієнтовний час приготування;
- позначку “улюблена страва”.

Після успішного редагування система повинна:

- зберегти оновлені дані;
- оновити дату останнього редагування;
- відобразити змінені значення у каталозі страв;
- використовувати оновлені дані в тижневому плані та списку продуктів.

### FR-05. Видалення страви

Система повинна дозволяти користувачу видалити страву з каталогу.
Перед видаленням будь-якої страви система повинна запитати підтвердження дії.
Якщо страва була додана до тижневого плану, перед видаленням система повинна показати попередження, що страва буде видалена з каталогу та прибрана з усіх відповідних прийомів їжі. Після підтвердження користувача страва видаляється з каталогу, локального сховища та всіх записів тижневого плану.

### FR-06. Додавання страви до тижневого плану

Користувач повинен мати можливість обрати будь-яку страву з каталогу незалежно від її типу прийому їжі.
Якщо тип страви не відповідає обраному типу прийому їжі, система повинна дозволити додавання, але показати попередження.

Користувач повинен мати можливість обрати:

- день тижня;
- тип прийому їжі;
- страву з каталогу.

Після додавання:

- страва відображається у тижневому плані;
- система оновлює список продуктів;
- система перевіряє правила сумісності для обраного дня.

### FR-07. Заміна страви в тижневому плані

Система повинна дозволяти користувачу змінити страву, яка вже додана до певного прийому їжі, шляхом видалення поточної страви з плану та додавання нової страви в порожню комірку.

Після видалення поточної страви:
- комірка плану стає порожньою;
- список продуктів автоматично перераховується;
- правила сумісності перевіряються повторно.

Після додавання нової страви:
- у плані відображається нова страва;
- список продуктів автоматично перераховується;
- правила сумісності перевіряються повторно.

### FR-08. Видалення страви з тижневого плану

Система повинна дозволяти користувачу прибрати страву з конкретного прийому їжі в плані.

Після видалення:

- комірка плану стає порожньою;
- список продуктів оновлюється;
- якщо після видалення порушення сумісності зникає, попередження більше не відображається.

### FR-09. Пошук страв

Система повинна надавати користувачу можливість шукати страви за назвою.

Пошук має виконуватись за частковим входженням тексту.

Пошук не повинен залежати від регістру літер.

### FR-10. Фільтрація страв

Система повинна дозволяти фільтрувати каталог страв за такими параметрами:

- тип прийому їжі;
- тег;
- рівень складності;
- улюблені страви;
- максимальний час приготування.

### FR-11. Сортування страв

Система повинна дозволяти сортувати каталог страв за:

- назвою;
- часом приготування;
- датою створення;
- рівнем складності.

### FR-12. Формування списку продуктів

Система повинна автоматично формувати список продуктів на основі страв, доданих до тижневого плану.

У списку продуктів мають відображатися:

- назва продукту;
- загальна кількість;
- одиниця вимірювання;
- категорія продукту.

Якщо однаковий продукт використовується у кількох стравах з однаковою одиницею вимірювання, система повинна об’єднати його кількість в один запис.

### FR-13. Перевірка простих правил сумісності

Система повинна перевіряти прості правила сумісності страв у межах одного дня.

Система повинна показувати попередження, якщо:

- одна й та сама страва додана в один день більше одного разу;
- тип страви не відповідає типу прийому їжі, до якого її додано;
- в одному дні обрано більше двох страв із тегом “важке”;
  
Попередження не повинно блокувати збереження плану, але має бути видимим для користувача.

### FR-14. Відображення тижневого плану

Система повинна відображати тижневий план у вигляді таблиці або карток.

У плані мають бути показані:

- дні тижня;
- типи прийомів їжі;
- обрані страви;
- порожні комірки, якщо страва ще не вибрана;
- попередження про несумісність, якщо вони є.

### FR-15. Відображення підсумкової статистики

Система повинна відображати коротку статистику по плану харчування, зокрема:

- кількість створених страв;
- кількість заповнених прийомів їжі на тиждень;
- кількість порожніх прийомів їжі;
- кількість улюблених страв;
- кількість попереджень сумісності;
- загальну кількість позицій у списку продуктів.

### FR-16. Збереження даних між сесіями

Після перезавантаження сторінки збережені страви та тижневий план не повинні втрачатися, 
якщо користувач не видалив їх явно або не очистив localStorage браузера.

### FR-17. Очищення фільтрів

Система повинна надавати користувачу можливість одним натисканням скинути всі активні фільтри та повернутися до повного каталогу страв.

### FR-18. Скидання тижневого плану

Система повинна дозволяти користувачу очистити тижневий план без видалення каталогу страв.

Перед очищенням система повинна запитати підтвердження дії.

Після підтвердження:

- усі обрані страви прибираються з плану;
- список продуктів очищується;
- каталог страв залишається без змін.

### FR-19. Відновлення demo data

Перед завантаженням demo data система повинна показати користувачу попередження, що поточні дані, зокрема створені страви, інгредієнти, теги та тижневий план, будуть видалені та замінені демонстраційними даними.

Після підтвердження користувача система повинна очистити поточні дані та завантажити demo data.

Після запуску demo data користувач повинен бачити:
- приклади страв;
- приклади інгредієнтів;
- приклади тегів;
- частково заповнений тижневий план.

## 6. Бізнес-правила

### BR-01. Обов’язкові поля страви

Під час створення страви обов’язковими є такі поля:

- назва;
- тип прийому їжі;
- хоча б один інгредієнт;
- рівень складності;
- орієнтовний час приготування.

### BR-02. Назва страви

Назва страви:

- повинна містити від 3 до 80 символів;
- не повинна складатися лише з пробілів;
- не повинна повністю дублювати назву іншої страви в каталозі.

### BR-03. Опис страви

Опис страви:

- є необов’язковим;
- якщо заповнений, повинен містити не більше 500 символів.

### BR-04. Тип прийому їжі

Для страви допускаються лише такі значення типу прийому їжі:

- Breakfast;
- Lunch;
- Dinner;
- Snack.

### BR-05. Рівень складності

Для страви допускаються лише такі значення рівня складності:

- Easy;
- Medium;
- Hard.

### BR-06. Час приготування

Орієнтовний час приготування:

- є обов’язковим;
- повинен бути цілим числом;
- допустимий діапазон: від 5 до 240 хвилин.

### BR-07. Інгредієнти

Для однієї страви:

- має бути вказаний хоча б один інгредієнт;
- можна вказати не більше 20 інгредієнтів;
- назва інгредієнта повинна містити від 2 до 50 символів;
- кількість інгредієнта повинна бути більшою за 0;
- одиниця вимірювання не може бути порожньою.

### BR-08. Одиниці вимірювання

Для інгредієнтів допускаються такі одиниці вимірювання:

- г;
- кг;
- мл;
- л;
- шт;
- ч. л.;
- ст. л.;
- за смаком.

### BR-09. Теги

Для однієї страви:

- можна вказати не більше 6 тегів;
- теги не повинні дублюватися в межах однієї страви;
- один тег повинен містити від 1 до 20 символів;
- тег не повинен складатися лише з пробілів.

### BR-10. Додавання страви до плану

В одному дні для одного типу прийому їжі може бути обрана тільки одна основна страва.

### BR-11. Повторення страви в один день

Одна й та сама страва не повинна повторюватися в межах одного дня.

Якщо користувач додає однакову страву повторно в той самий день, система повинна показати попередження.

### BR-12. Правило “важких” страв

Якщо в одному дні обрано більше двох страв із тегом “важке”, система повинна показати попередження про перевантажений день.

### BR-13. Правило типу прийому їжі

Рекомендована відповідність типів:
Breakfast — сніданок;
Lunch — обід;
Dinner — вечеря;
Snack — перекус.

Система не повинна блокувати додавання страви до невідповідного типу прийому їжі.
Якщо користувач додає страву до прийому їжі, який не відповідає типу страви, система повинна показати попередження, але зберегти вибір користувача.

### BR-14. Список продуктів

Якщо однакові інгредієнти мають однакову назву та одиницю вимірювання, система повинна об’єднати їх кількість у списку продуктів.

Якщо однакові продукти мають різні одиниці вимірювання, система повинна показати їх окремими рядками.

### BR-15. Видалення страви

Будь-яка страва може бути видалена тільки після явного підтвердження користувача.

### BR-16. Очищення плану

Очищення тижневого плану не повинно видаляти створені страви з каталогу.

### BR-17. Локальне збереження

Усі дані застосунку зберігаються тільки в браузері поточного користувача.

Якщо користувач очистить localStorage браузера, дані можуть бути втрачені.

## 7. Правила відображення станів

### SR-01

Якщо в каталозі немає жодної страви, система повинна показати порожній стан із текстом, що користувач може створити першу страву або завантажити demo data.

### SR-02

Якщо тижневий план порожній, система повинна показати повідомлення, що план ще не заповнений.

### SR-03

Якщо для дня є попередження сумісності, система повинна візуально позначити цей день.

### SR-04

Страви, позначені як улюблені, повинні мати окрему візуальну позначку в каталозі.

### SR-05

Активні фільтри повинні бути помітні користувачу, щоб було зрозуміло, чому в каталозі відображаються не всі страви.

### SR-06

Порожні прийоми їжі в тижневому плані повинні відображатися як доступні для додавання страви.

## 8. Нефункціональні вимоги

### NFR-01. Тип застосунку

Система повинна бути реалізована як вебзастосунок, доступний через браузер.

### NFR-02. Сумісність із GitHub Pages

Система повинна працювати без обов’язкової серверної частини та бути придатною для розгортання на GitHub Pages.

### NFR-03. Сумісність із браузерами

Система повинна коректно працювати в актуальних версіях:

- Google Chrome;
- Mozilla Firefox;
- Microsoft Edge;
- Safari.

### NFR-04. Адаптивність

Інтерфейс повинен бути придатним для використання щонайменше на таких розмірах екрана:

- 1920×1080;
- 1366×768;
- 390×844.

### NFR-05. Продуктивність

Після відкриття застосунку каталог страв і тижневий план повинні ставати доступними для взаємодії не довше ніж за 2 секунди за наявності до 100 страв і до 28 записів у плані.

### NFR-06. Збереження даних

Після перезавантаження сторінки збережені страви, тижневий план не повинні втрачатися, якщо користувач не видалив їх явно або не очистив localStorage браузера.

### NFR-07. Зрозумілість повідомлень

Усі повідомлення про помилки валідації повинні бути сформульовані зрозуміло і прямо вказувати, що саме введено неправильно.

### NFR-08. Надійність

У разі відсутності даних у локальному сховищі система повинна коректно відкриватися і показувати порожній стан без критичної помилки.

### NFR-09. Зручність використання

Користувач повинен мати можливість додати страву до плану не більше ніж за 4 основні дії після відкриття застосунку, якщо страва вже існує в каталозі.

### NFR-10. Безпека даних

Система не повинна вимагати введення паролів, платіжних даних, медичних даних або іншої чутливої інформації.

### NFR-11. Доступність

Основні кнопки, поля форм і фільтри повинні мати зрозумілі підписи, щоб користувач міг користуватися застосунком без додаткової інструкції.

## 10. Критерії приймання для ключових функцій

### AC-01. Створення валідної страви

Передумова: користувач відкрив форму створення страви.

Умова: користувач заповнив усі обов’язкові поля коректними даними.

Очікуваний результат: після натискання кнопки збереження нова страва додається до каталогу, має унікальний ID і стає доступною для додавання в тижневий план.

### AC-02. Заборона створення невалідної страви

Передумова: користувач відкрив форму створення страви.

Умова: хоча б одне обов’язкове поле порожнє або містить невалідне значення.

Очікуваний результат: страва не створюється, система показує повідомлення про помилку, введені користувачем дані не зникають із форми.

### AC-03. Додавання страви до плану

Передумова: у каталозі існує хоча б одна страва.

Умова: користувач обирає день тижня, тип прийому їжі та страву.

Очікуваний результат: обрана страва відображається у відповідній комірці тижневого плану, а список продуктів автоматично оновлюється.

### AC-04. Заміна страви в заповненому прийомі їжі

Передумова: у певному дні та прийомі їжі вже є страва.

Умова: користувач видаляє поточну страву з цього прийому їжі, а потім додає нову страву в порожню комірку.

Очікуваний результат: стара страва більше не відображається у цьому прийомі їжі, нова страва відображається у відповідній комірці, список продуктів перераховується, а тижневий план показує оновлені дані.

### AC-05. Формування списку продуктів

Передумова: у тижневому плані є кілька страв з інгредієнтами.

Умова: користувач відкриває список продуктів.

Очікуваний результат: система показує список усіх продуктів із плану, однакові продукти з однаковими одиницями вимірювання об’єднані в один запис.

### AC-06. Перевірка повторення страви

Передумова: у певному дні вже є страва “Омлет”.

Умова: користувач додає “Омлет” до іншого прийому їжі в той самий день.

Очікуваний результат: система показує попередження, що страва повторюється в межах одного дня.

### AC-07. Перевірка правила “важких” страв

Передумова: у системі є страви з тегом “важке”.

Умова: користувач додає більше двох таких страв в один день.

Очікуваний результат: система показує попередження про перевантажений день.

### AC-08. Пошук страви

Передумова: у каталозі є страви з різними назвами.

Умова: користувач вводить частину назви страви в поле пошуку.

Очікуваний результат: у каталозі залишаються тільки страви, назва яких містить введений текст.

### AC-09. Фільтрація за тегом

Передумова: у каталозі є страви з різними тегами.

Умова: користувач обирає фільтр за тегом, наприклад “швидко”.

Очікуваний результат: у каталозі відображаються тільки страви з відповідним тегом.

### AC-10. Сортування за часом приготування

Передумова: у каталозі є щонайменше 3 страви з різним часом приготування.

Умова: користувач обирає сортування за часом приготування за зростанням.

Очікуваний результат: страви відображаються від найшвидшої до найдовшої за часом приготування.

### AC-11. Збереження між сесіями

Передумова: користувач створив одну або більше страв і додав хоча б одну страву до плану.

Умова: користувач перезавантажує сторінку або повторно відкриває застосунок.

Очікуваний результат: раніше створені страви та тижневий план залишаються в системі.

### AC-12. Очищення тижневого плану

Передумова: у тижневому плані є додані страви.

Умова: користувач натискає кнопку очищення плану та підтверджує дію.

Очікуваний результат: усі прийоми їжі в плані стають порожніми, список продуктів очищується, але каталог страв залишається без змін.

### AC-13. Відновлення demo data

Передумова: користувач відкрив застосунок і в системі можуть бути вже створені користувачем дані.

Умова: користувач натискає кнопку завантаження demo data та підтверджує дію.

Очікуваний результат: система видаляє поточні дані користувача, зокрема створені страви, інгредієнти, теги та тижневий план, після чого завантажує демонстраційні страви, інгредієнти, теги та частково заповнений тижневий план.

### AC-14. Попередження про невідповідний тип прийому їжі

Передумова: у каталозі існує страва з типом Lunch.

Умова: користувач додає цю страву до прийому їжі Breakfast.

Очікуваний результат: страва додається до сніданку, список продуктів оновлюється, а система показує попередження про невідповідність типу прийому їжі.

## 11. Припущення та обмеження

- Система не використовує реальні персональні, медичні або платіжні дані.
- Система не надає професійних дієтичних або медичних рекомендацій.
- Система не потребує серверної частини для базової роботи.
- Усі дані зберігаються локально в браузері поточного користувача.
- Якщо користувач очистить localStorage браузера, дані можуть бути втрачені.
- Система розрахована на навчальний сценарій використання, а не на промислову багатокористувацьку експлуатацію.
- Система не гарантує точний підрахунок калорій, білків, жирів, вуглеводів або інших нутрієнтів.
- Список продуктів формується тільки на основі інгредієнтів, які користувач сам додав до страв.
- Правила сумісності є простими підказками для планування, а не обов’язковими медичними обмеженнями.
- Застосунок має бути придатний для розгортання на GitHub Pages, тому вся основна логіка повинна працювати на клієнтській стороні.
